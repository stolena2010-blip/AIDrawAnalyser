# המלצות לשיפור חווית משתמש — AIDrawAnalyser

> מטרת המסמך: לשפר את חווית המשתמש של AIDrawAnalyser כך שהאפליקציה תרגיש פחות כמו כלי פנימי ויותר כמו מוצר מקצועי שאפשר להציג, למכור ולהטמיע אצל לקוח.

---

## סיכום מנהלים

הממשק הנוכחי שימושי וחזק, במיוחד למשתמש טכני שמבין שרטוטים ותהליכי ייצור. יש זרימה בסיסית טובה: העלאת קובץ, ניתוח, תוצאות, ייצוא. מצב המכלולים מתקדם יחסית וכולל ניווט בין שרטוטים, ניתוח קשרים ודוחות.

עם זאת, מבחינת מכירה והטמעה אצל לקוח חדש, הממשק עדיין מרגיש עמוס, טכני, ומעט “פנימי”. יש הרבה מידע בבת אחת, הרבה אייקונים וצבעים, וחלק מהמידע שמעניין מנהל מערכת מופיע קרוב מדי לחווית המשתמש הרגילה.

**ציון UX נוכחי:** 84/100  
**יעד UX למכירה:** 92+/100

הדרך לשם אינה דורשת בנייה מחדש של האפליקציה. צריך בעיקר לארגן מחדש את המסכים, להבליט את מה שחשוב למשתמש, להסתיר debug/admin כברירת מחדל, ולהוסיף שלב review לפני יצוא.

---

## עקרונות UX מומלצים

### 1. המשתמש רוצה תשובה, לא raw data

המשתמש העסקי לא רוצה קודם לראות JSON, מודלים, טוקנים, וכל שדות הפלט. הוא רוצה לדעת:

- מה זוהה בשרטוט?
- מה בטוח יחסית?
- מה דורש בדיקה?
- איזה דוח אפשר להוריד?

### 2. AI צריך להרגיש כמו עוזר, לא כמו מקור אמת מוחלט

בשרטוטים הנדסיים תמיד צריך להשאיר מקום לבדיקה אנושית. לכן החוויה צריכה להדגיש review, warnings ותיקון לפני export.

### 3. הממשק צריך להתאים לשני קהלים

- **משתמש רגיל:** רוצה לנתח שרטוט ולהפיק דוח.
- **מנהל / מפתח / בעלת המערכת:** רוצה לראות מודלים, עלויות, cache, JSON, prompt/debug.

היום שני הקהלים מעורבבים חלקית. ההפרדה ביניהם תעלה מאוד את תחושת המקצועיות.

## בעיות UX מרכזיות במצב הנוכחי

### 1. עומס מידע אחרי ניתוח

לאחר ניתוח מוצגים הרבה סעיפים: כרטיס שרטוט, BOM, תהליכים, ציפויים, צביעות, תקנים, אריזה, הערות, warnings, JSON וייצוא.

**השפעה:** משתמש חדש עלול לא להבין מה הדבר החשוב ביותר ומה דורש פעולה.

### 2. Debug גלוי מדי

מידע כמו JSON מלא, מודל בפועל, טוקנים ועלויות מפורטות חשוב מאוד, אבל לא צריך להיות חלק מרכזי מהחוויה של המשתמש הרגיל.

**השפעה:** לקוח פוטנציאלי עלול להרגיש שהוא רואה כלי ניסיוני ולא מוצר מוגמר.

### 3. אין שלב Review מסודר

כרגע המשתמש מקבל תוצאה ואז מייצא. חסר שלב שבו הוא יכול לאשר או לתקן שדות קריטיים לפני יצירת PDF/Excel.

**השפעה:** במוצר AI לתעשייה זה מוריד אמון, כי משתמשים יודעים שהמודל עלול לטעות.

### 4. ה-flow במצב מכלולים ארוך

מצב המכלולים חזק, אבל יש בו הרבה פעולות: העלאה, ניתוח, דפדוף, ניתוח קשרים, הורדת דוחות. זה נכון פונקציונלית, אבל כדאי להפוך אותו ליותר guided.

**השפעה:** משתמש חדש עלול לפספס את הצעד הבא או לא להבין מתי התוצאה “מוכנה”.

### 5. עיצוב צבעוני מדי למוצר B2B

יש שימוש רחב באייקונים, gradients, צבעי מצב, expanderים וטקסטים ארוכים. זה נחמד לכלי פנימי, אבל מוצר תעשייתי למכירה צריך להרגיש יותר שקט, יציב וצפוי.

**השפעה:** פחות תחושת enterprise / production.

## שיפורים בעדיפות גבוהה

### 1. מסך תוצאות מקוצר בראש

לאחר ניתוח, להציג בראש המסך “תקציר החלטה” לפני כל הפירוט.

**מבנה מומלץ:**

| שדה | תוכן |
|---|---|
| Part Number | הערך שזוהה |
| Drawing Number | הערך שזוהה |
| Revision | הערך שזוהה |
| Customer | הערך שזוהה |
| Material | הערך שזוהה |
| Role | PART / ASSEMBLY |
| Warnings | מספר אזהרות לפי חומרה |
| Export | כפתורי PDF / Excel |

**למה זה חשוב:** המשתמש מקבל תשובה תוך 5 שניות ולא צריך לגלול כדי להבין מה קרה.

### 2. Tabs לתצוגת תוצאות

במקום גלילה ארוכה, לחלק את התוצאה לטאבים:

- **סקירה** — שדות מרכזיים, warnings, סטטוס ניתוח.
- **BOM** — טבלת חלקים.
- **תהליכים** — עיבוד, ריתוך, חום, ציפוי, צבע, בדיקות.
- **תקנים** — standards עם סימון חשודים.
- **אריזה והערות** — packaging / notes.
- **ייצוא** — PDF / Excel / JSON.
- **Debug** — JSON, מודל, טוקנים, רק במצב מנהל.

**למה זה חשוב:** מפחית עומס קוגניטיבי ומרגיש יותר כמו מוצר מסודר.

### 3. שלב Review לפני Export

להוסיף מצב עריכה לשדות חשובים לפני הפקת דוחות.

**שדות מומלצים לעריכה:**

- Part Number
- Drawing Number
- Revision
- Customer
- Material
- Quantity
- Standards
- Coating / Painting rows
- Packaging notes

**התנהגות מומלצת:**

- שדות עם warning מסומנים בצבע עדין.
- ליד כל שדה מופיע מקור: AI / OCR / Filename / BOM cross-reference.
- כפתור “אשר תוצאה” לפני export.
- הדוח מציין אם עבר review ידני.

**למה זה חשוב:** זה השיפור הכי משמעותי לאמון במוצר AI.

### 4. הפרדת מצב משתמש ומצב מנהל

להגדיר שני modes:

**משתמש רגיל רואה:**
- העלאה
- תוצאות
- warnings
- review
- export

**מנהל רואה בנוסף:**
- מודל בפועל לכל שלב
- עלויות וטוקנים
- JSON מלא
- cache
- בחירת מודלים
- fallback settings

**למה זה חשוב:** לקוח עסקי לא צריך לראות את כל הקרביים של המערכת.

### 5. Demo Mode למכירה

להוסיף כפתור “טען דוגמה” במסך הראשון.

**מה הוא עושה:**
- טוען תוצאת JSON מוכנה מראש.
- מציג תוצאה מלאה בלי Azure ובלי קובץ PDF אמיתי.
- מאפשר להראות PDF/Excel לדוגמה.

**למה זה חשוב:** בדמו מכירה אין זמן לתקלות API, קבצים כבדים, או המתנה של 40 שניות.

## שיפורים בעדיפות בינונית

### 6. Progress מפורט בזמן ניתוח

במקום spinner כללי, להציג שלבים:

1. בודק קובץ PDF
2. ממיר עמודים לתמונות
3. קורא שדות בסיסיים
4. קורא תהליכים ותקנים
5. מריץ OCR fallback אם צריך
6. מריץ ולידציות
7. מכין תצוגת תוצאות

במצב מכלולים להראות גם:

- כמה שרטוטים הסתיימו
- כמה נכשלו
- איזה קובץ מעובד כרגע
- האם Overview Image זוהתה

### 7. Empty State מקצועי במסך הראשון

לפני העלאת קובץ, המסך צריך להסביר בקצרה מה יקרה.

**מבנה מומלץ:**

- כותרת קצרה: “נתח שרטוט PDF והפק דוח ייצור”
- שלושה bullets:
  - חילוץ P/N, חומר, תקנים ותהליכים
  - אזהרות על ערכים חשודים
  - יצוא ל-PDF/Excel
- אזור upload ברור
- כפתור Demo

### 8. שיפור אזהרות

Warnings צריכים להיות actionable.

במקום רק להציג הודעה טכנית, כל warning צריך לענות:

- מה חשוד?
- למה זה חשוד?
- מה המשתמש צריך לבדוק?
- האם זה חוסם export או רק המלצה?

**סיווג מומלץ:**

| חומרה | משמעות UX |
|---|---|
| Critical | דורש בדיקה לפני export |
| High | מומלץ לבדוק |
| Medium | שווה תשומת לב |
| Low / Info | מידע בלבד |

### 9. איחוד שפה ומיקרו-קופי

כדאי לבחור סגנון אחיד לכל הכפתורים והכותרות.

**דוגמאות:**

- “נתח שרטוט” במקום “נתח PDF” / “נתח קובץ” לסירוגין.
- “צור דוח PDF” במקום “PDF מלא” במקומות מסוימים.
- “בדיקה נדרשת” במקום הודעות טכניות מדי.

### 10. התאמת עיצוב למכירה B2B

המלצה עיצובית:

- להפחית gradients.
- להשתמש בפלטה רגועה: כחול כהה, אפור, ירוק הצלחה, כתום אזהרה, אדום שגיאה.
- לצמצם emoji בכותרות עסקיות.
- להשאיר אייקונים רק לכפתורים או מצבי סטטוס.
- לשמור על טבלאות צפופות וברורות.

## שיפורים בעדיפות נמוכה

### 11. Onboarding קצר

להוסיף expander קטן “איך זה עובד” במסך הראשון.

לא מדריך ארוך, רק 3 שלבים:

1. מעלים PDF.
2. המערכת מחלצת שדות ותהליכים.
3. בודקים warnings ומייצאים דוח.

### 12. שמירת היסטוריית ניתוחים

מסך “ניתוחים אחרונים” יכול לעזור מאוד למשתמש חוזר.

**שדות:**

- תאריך
- שם קובץ
- P/N
- סטטוס
- מספר warnings
- כפתור פתיחה / הורדה

### 13. תצוגת השוואה בין שרטוטים במכלול

במצב assembly, להוסיף טבלת overview לכל הקבצים שנותחו:

| # | File | P/N | Role | Material | Warnings | Status |
|---|---|---|---|---|---|---|

לחיצה על שורה תפתח את כרטיס השרטוט.

### 14. Export Preview

לפני יצירת PDF/Excel, להציג “מה ייכלל בדוח”.

זה יכול למנוע הפקת קבצים מיותרים ולתת תחושת שליטה.

## המלצה מסכמת

לא צריך להחליף את Streamlit כרגע. הממשק הקיים מספיק טוב כבסיס, וההשקעה הנכונה היא ארגון וחידוד החוויה.

המהלך הכי משתלם הוא:

1. להוסיף Demo Mode.
2. לבנות Summary + Warnings בראש התוצאות.
3. להפריד Debug/Admin ממשתמש רגיל.
4. להוסיף Review/Edit לפני export.

ארבעת השינויים האלה יהפכו את האפליקציה להרבה יותר נוחה להצגה לקונה, בלי להיכנס לריפקטור גדול או החלפת טכנולוגיה.
