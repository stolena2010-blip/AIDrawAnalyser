## מה חובה לסדר לפני מכירה

### 1. לנקות ולסנכרן תיעוד

**בעיה:** חלק מהמסמכים עדיין מתארים מצב ישן של הפרויקט, למשל `extractor.py` שכבר אינו קיים, או ציון ישן יותר.

**המלצה:**
- לעדכן את README כך שישקף בדיוק את המצב הנוכחי.
- לעדכן את `IMPROVEMENT_RECOMMENDATIONS.md` או להעביר אותו ל-archive.
- ליצור מסמך `TECHNICAL_DUE_DILIGENCE.md` שמסביר ארכיטקטורה, תלויות, סיכונים ומגבלות.

**חשיבות למכירה:** קונה בודק עקביות. תיעוד לא מעודכן מוריד אמון גם אם הקוד טוב.

### 2. לנעול תלויות

**בעיה:** `requirements.txt` משתמש בגרסאות פתוחות עם `>=`, ולכן התקנה עתידית עלולה להביא גרסה שוברת.

**המלצה:**
- ליצור `requirements.lock.txt` מתוך הסביבה שעובדת בפועל.
- להשאיר `requirements.txt` כקובץ development רך.
- לתעד איזו גרסת Python נתמכת רשמית.

**חשיבות למכירה:** קונה רוצה לדעת שהמוצר ניתן לשחזור במחשב נקי.

### 3. להפוך lint לחוסם ב-CI

**בעיה:** ה-CI מריץ ruff, אבל לא חוסם כשל כי מוגדר `continue-on-error: true`.

**המלצה:**
- להוסיף קונפיגורציית ruff מסודרת.
- לתקן שגיאות lint קיימות.
- להפוך lint לחובה.

**חשיבות למכירה:** נותן תחושת מוצר מתוחזק ולא סקריפט ניסיוני.

### 4. להפריד נתוני לקוחות ודוגמאות רגישות

**בעיה:** תיקיות כמו `draws/`, `output/`, `REPORTS/` מוחרגות ב-gitignore, אבל צריך לוודא שאין מידע רגיש שכבר נכנס להיסטוריה.

**המלצה:**
- להריץ בדיקה שאין קבצי לקוח tracked.
- ליצור סט דוגמאות sanitized להדגמה.
- להכין `sample_drawings/` עם קבצים שאין בהם IP של לקוחות.

**חשיבות למכירה:** קונה יחשוש מחשיפה משפטית אם יש שרטוטים אמיתיים או נתוני לקוחות.

### 5. להכין Demo Pack

צריך חבילת הדגמה שמראה את הערך תוך 5 דקות.

**תכולה מומלצת:**
- 3 שרטוטים לדוגמה: פשוט, מורכב, assembly.
- פלט Excel לדוגמה.
- דוח PDF לדוגמה.
- צילום מסך של UI.
- מסמך קצר: לפני/אחרי, כמה זמן נחסך, אילו טעויות נמנעו.

**חשיבות למכירה:** רוב הקונים לא יקראו קוד בהתחלה. הם רוצים לראות תוצאה.

## שיפורים שמעלים את ערך המכירה

### 1. מדידת דיוק על סט בדיקה אמיתי

לבנות טבלה של 20-50 שרטוטים עם ground truth ידני:

| שדה | מדד |
|---|---|
| P/N | אחוז זיהוי נכון |
| Revision | אחוז זיהוי נכון |
| Material | אחוז זיהוי נכון |
| Standards | Precision / Recall |
| Coating / Painting | Precision / Recall |
| BOM Items | אחוז התאמה |

**יעד טוב למכירה:** להראות 85%-95% דיוק בשדות מרכזיים, עם הסבר איפה נדרש review אנושי.

### 2. להוסיף מצב Review אנושי

האפליקציה לא צריכה להבטיח 100% אוטומציה. למכירה B2B עדיף להציג אותה ככלי שמאיץ review.

**המלצה:**
- לסמן שדות confidence / warnings.
- לאפשר למשתמש לתקן תוצאה לפני export.
- לשמור מי תיקן ומה תוקן.

**ערך עסקי:** הופך את האפליקציה לכלי עבודה אמין יותר ולא “קופסה שחורה”.

### 3. להוסיף Export מותאם ERP

קונים תעשייתיים ירצו לחבר את התוצאה למערכות קיימות.

**המלצה:**
- להגדיר schema יציב ל-JSON.
- ליצור CSV/Excel בפורמט שניתן לייבוא.
- לתעד mapping לשדות ERP/MES.

### 4. לשפר התקנה ופריסה

**אפשרויות:**
- חבילת Windows עם סקריפט התקנה ברור.
- Docker image לפריסה פנימית.
- גרסת cloud עם Azure App Service.

**המלצה מסחרית:** אם מוכרים לחברות ביטחוניות/ייצור, להתחיל מ-on-prem / private deployment, לא SaaS ציבורי.

### 5. להוסיף Audit Log

לקונה עסקי חשוב לדעת מה נותח, מתי, על ידי מי, ובאיזה מודל.

**המלצה:**
- לשמור run id.
- לשמור model/version/prompt version.
- לשמור user/action/export.
- לא לשמור קבצים רגישים בלי הגדרה מפורשת.
